# J-AP Plus 极简工作流重构计划 (Streamlined Workflow Plan)

> **目标**：剥离繁文缛节，将 J-AP Plus 从“填表式八股文生成器”重构为“极客向、按需定制、直达代码的 AI 结对设计副驾驶”。

---

## 1. 当前系统存在的“鸡肋”问题总结

在现有的 01 到 08 的瀑布流式（Phase-locked）设计流转中，暴露出以下拖慢节奏、且产出不高的环节：

1. **强迫全家桶（过度设计）**：不论项目大小，强制生成 02(数据库)、03(状态机)、06(前端页面)。对于纯 API 模块或纯前端静态页而言，这是纯粹的时间与算力浪费。
2. **08_SDD 缺乏实操性**：目前 08_SDD 动辄几千字，是对 01-07 的冗余复述。程序员需要的是“要建什么表”、“要写什么接口”的 Checkbox，而不是一篇论文。
3. **05_验收测试 形同虚设**：在代码未动之前，强迫生成 BDD 测试用例，用户体验极差，且指导意义薄弱。
4. **问卷式需求收集僵化**：前置的 Elicitation 问卷限制了用户的发散性思维，不符合现代开发者边聊边构思（Explore）的习惯。

---

## 2. 核心重构方案 (The New Workflow)

将现有的 **8步瀑布流** 缩减重构为 **3段式敏捷流 (Idea -> Design -> Action)**：

### Phase 1: 智能自适应的探索与立项 (Adaptive Explore & Propose)
- **双轨制需求收集**：在入口处，提供两种模式供用户选择：
  1. **沙盒对话模式 (Explore)**：适合有清晰思路的资深开发者。用户在一个自由的对话框中描述需求，随时点击“将当前讨论固化为需求草案”，直接立项进入 `01_需求草案`。
  2. **智能向导模式 (Guided Questionnaire)**：保留问卷形式，照顾“不知道从何说起”的新手用户。但**取消冗长的固定题库**，改为大模型动态出题。
- **大模型智能判停 (Auto-Finalize)**：在智能向导模式下，大模型会根据用户的回答质量进行自我评估。当它认为收集到的信息已经足够生成一份完整的 01 需求时，**主动结束问卷**，不再死板地抛出剩下的问题，直接进入设计环节。

### Phase 2: 按需设计 (A la Carte Design)
- 01 生成后，**系统弹出“架构选配模块”**（Checkbox）：
  - [ ] 02_数据库表结构
  - [ ] 03_核心业务状态机
  - [ ] 04_RESTful API 契约
  - [ ] 06_前端页面与组件树
- 彻底废除 `05_验收测试` 作为核心节点的强制性。
- 后端根据用户的勾选，利用 **DAG 并发引擎** 同时生成选中的文件，时间压缩至原来的 1/3。

### Phase 3: 直达代码的 Action (Tasks Generation)
- 将现有的 `08_SDD` 重命名并重构为 `08_Actionable_Tasks (开发任务清单)`。
- 大模型不再输出长篇大论，而是严格输出带有复选框 (`- [ ]`) 的具体编码任务（细化到文件路径和函数签名）。
- **产物可以直接丢给 Cursor/Trae 等 AI 编辑器自动执行编码。**

---

## 3. 具体修改步骤 (Technical Steps)

为了实现上述方案，我们需要在现有 Node.js + React 代码底座上进行如下改造：

### Step 1: 状态机引擎解耦 (State Machine Decoupling)
- **修改 `meta.json` 结构**：引入 `selectedModules: string[]` 字段。
- **重构 `taskService.ts` 校验逻辑**：不再硬编码校验 `01-07` 是否全为 `APPROVED`。改为：基于 `selectedModules` 动态判断前置节点是否就绪。

### Step 2: 编排层并发改造 (Concurrent Orchestration)
- **重构 `generate-base-next` 接口**：取消 02-07 的严格顺延逻辑。
- 引入 `Promise.all`。当 01 完成后，若用户勾选了 02、04，则并行向大模型发起两次生成请求，极大缩短等待黑盒时间。

### Step 3: Prompt 重构 (LLM Engineering)
- **剔除 05 测试用例 Prompt**。
- **重写 08 的 Prompt**：强行注入 `tasks.md` 风格指令。要求输出格式为：
  ```markdown
  ## 1. 数据库构建
  - [ ] `schema.sql`: 执行建表语句 (User, Order)
  ## 2. API 接口实现
  - [ ] `src/api/user.ts`: 实现 GET /users 接口逻辑
  ```

### Step 4: 前端交互适配 (UI/UX Refactoring)
- 在 01_APPROVED 后，拦截下一步，弹出“**请选择需要生成的架构模块**”对话框。
- 右侧文件列表动态渲染：未勾选的模块直接置灰或隐藏，不再显示为 `PENDING`。
- 将 08 的标题变更为“生成开发任务清单”。

如果老板您确认该计划，我们将分为三个 Sprint 快速落地，保证系统随时可运行（不破坏现有数据）：

- **Sprint 1（拆骨）**：完成 **Step 1 & Step 3**。先从底层解除 01-07 的强绑定，把 08_SDD 的 Prompt 改写为 Tasks 风格。这步做完，立刻就能感受到产物变得“干练”、“有用”。
