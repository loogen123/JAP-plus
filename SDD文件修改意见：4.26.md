# SDD文件修改意见：4.26

## 目标

将 JAP-plus 从“生成复杂设计文档”调整为“生成可指导开发 Agent 的轻量执行资料包”。

这些文件合起来要满足 SDD 约束目标：需求清楚、名词统一、规则明确、能力边界清晰、执行有方向、验收可判断。

## 核心原则

1. 产物名全部使用中文。
2. 保留当前 `tasks/run-*` 单目录输出结构。
3. 不重做任务状态机、历史记录、WebSocket、审核流程。
4. 重点修改产物定义、Prompt、07 生成逻辑和前端文案。
5. 不在计划阶段写死数据库表、接口路径、组件名、目录结构。
6. 每个文件只写自己的职责范围，减少文件间矛盾。

## 建议产物

建议仍使用 01-07，降低代码改动成本：

```text
01_需求简报.md
02_领域词典.md
03_行为规则.md
04_能力意图.md
05_Agent执行指南.md
06_验收清单.md
07_SDD约束总览.md
```

## 文件职责

`01_需求简报.md`

只写目标、范围、用户角色、核心场景、不做什么。

`02_领域词典.md`

只写领域对象、名词解释、字段含义、状态词、业务术语。

`03_行为规则.md`

只写业务规则、状态变化、权限边界、异常情况。

`04_能力意图.md`

只写系统需要提供哪些能力，例如创建、查询、审核、导出。不要写死 REST 路径。

`05_Agent执行指南.md`

只写给开发 Agent 的执行顺序、阅读顺序、可自由设计范围、必须遵守的约束。

`06_验收清单.md`

只写验收条件、测试观察点、完成判断。不要写具体测试代码。

`07_SDD约束总览.md`

只写本批文件的约束关系：哪个文件是范围源头，哪个文件是术语源头，哪些内容不能新增需求。

## 修改范围

第一步：修改产物名称

修改 `src/constants/domainConstants.ts` 中的 `ARTIFACT_FILES`。

第二步：修改 Prompt

修改 `src/constants/promptTexts.ts` 和 `src/services/taskService.ts`。

删除或弱化这些约束：

```text
必须完整 OpenAPI
必须完整 Postman Collection
必须完整 HTML UI 原型
必须输出 Actionable Tasks Checklist
必须全局强一致
```

替换为：

```text
只输出当前文件职责范围内的内容
不要替开发 Agent 写死实现细节
不要新增其他文件职责范围的内容
只表达约束、意图、规则、验收
```

第三步：调整 07 生成逻辑

当前 07 是 `Actionable Tasks`，需要改为 `SDD约束总览`。

重点修改 `src/services/taskService.ts` 中针对 `fileId === "07"` 的特殊生成逻辑。

第四步：调整前端文案

修改 `public/index.html` 和 `public/js/main.js`。

建议文案：

```text
一键固化为需求草案 -> 生成 Agent 执行资料包
生成架构设计模块 -> 生成执行约束文件
生成任务清单 -> 生成 SDD 约束总览
导入前置设计直接生成 07 -> 导入历史资料生成约束总览
```

第五步：端到端验证

使用一个简单需求生成完整 01-07。

检查结果：

1. 每个文件是否只写自己的方向。
2. 是否明显比原来简洁。
3. 是否减少重复解释。
4. 是否没有写死数据库表、REST 路径、组件名。
5. 开发 Agent 是否能根据这些文件开始开发。

## 暂不建议修改

1. 暂不修改目录结构。
2. 暂不重构任务状态机。
3. 暂不新增复杂 SDD 校验器。
4. 暂不重构历史任务逻辑。
5. 暂不改 `tasks/run-*` 输出方式。

## 验收标准

1. 根目录下仍能正常生成一个 `tasks/run-*` 任务目录。
2. 任务目录内生成中文命名的 01-07 文件。
3. 文件内容轻量、明确、可指导 Agent。
4. 07 不再是开发任务清单，而是 SDD 约束总览。
5. 前端按钮和日志文案不再强调传统设计文档，而是强调 Agent 执行资料。
