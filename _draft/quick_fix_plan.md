# JAP Plus 快速稳定输出与质量提升计划 (Filewise 专属修正版)

## 一、核心目标

**优先级排序**：
1. **P0 - 绝对稳定**：彻底消灭因为 JSON 格式化、转义导致的生成失败（尤其是 02、04 等长文件）。全系统统一为唯一且最稳定的 Filewise 模式，砍掉容易失败的历史遗留“一键大满贯”模式。
2. **P1 - 质量与丰满度**：解决“内容太简略”的问题，放宽上下文截断限制，强化 Prompt 结构骨架。
3. **P2 - 所见即所得**：生成成功后，直接让用户在根目录看到带有传统业务命名（如 `01_产品功能脑图与用例.md`）的最终文件。

---

## 二、当前代码与原计划的冲突诊断

1. **原计划的“并行生成”不适用当前模式**：当前系统已切换至 `filewise`（单文件人工确认流水线），文件是用户点一下生成一个。原计划中的 05/06/07 并行在 `filewise` 模式下无法直接应用，且不是当前“出不来、内容少”的核心痛点。**结论：剔除并行逻辑，专注单文件质量与稳定。**
2. **导致失败的真凶是 JSON 结构化**：代码中大量使用 `invokeStructuredWithJsonFallback` 或让模型输出 `{"02_领域模型...": "..."}`。长 Markdown 文本塞进 JSON 极易因为换行符、引号导致解析崩溃。
3. **导致内容简略的真凶是硬截断**：`taskRoutes.ts` 中 `FILEWISE_CONTEXT_LIMIT` 首轮只有 5000 字符，重试更是降到 2400，长需求直接被腰斩。

## 三、修正后的快速修复方案 (Quick Fix)

### 3.1 彻底清理历史 legacy 模式，实现“架构极简”

**方案**：只保留最稳定的 `filewise` 单文件流水线，删掉之前容易失败的“一口气生成全部”接口和相关前端选项。
**做法**：
- 删除 `taskRoutes.ts` 中针对 `design-only` 遗留模式的大满贯生成逻辑。
- 前端 `index.html` 中彻底移除 `workflowMode` 切换开关，默认且强制进入流水线模式。

### 3.2 弃用结构化，改用纯文本直接输出（解决 P0 稳定）

**方案**：在 `filewise` 模式下，既然每次只生成一个文件，完全不需要模型返回 JSON 键值对。
**修改点**：`taskRoutes.ts` -> `filewiseGenerateCurrent` 链路。
**做法**：
- 修改系统提示词：`"You are generating a single file. Output ONLY the raw markdown/yaml/html content. DO NOT wrap it in JSON or any code blocks."`
- 调用模型时：直接 `const response = await model.invoke(...)`，取 `response.content` 即可，跳过所有 JSON Parse 逻辑。

### 3.2 提升上下文预算与 Prompt 骨架（引入 CoT 与结构化模板）

**方案**：释放上下文空间，逼迫模型写全细节，并引入 `improvement_plan_v3.md` 中的高级提示工程技术。
**修改点**：
1. **上下文预算调整**：首轮 `requirement` 上限提升至 **10000** 字符，二轮重试降级到 6000 字符。
2. **Chain of Thought (CoT) 与强制规范注入**：在 Prompt 中明确每种文件的生成步骤与严格格式（从 v3 计划中提取）：
   - **01 (产品脑图)**：强制输出 Mermaid graph TD 格式；必须包含至少 8 个核心用例的实现规格（API 序列、数据验证规则、错误码定义）。
   - **02 (领域模型)**：强制输出完整的 MySQL DDL 建表语句，包含主键、外键、索引 (`uk_`, `idx_`) 与业务约束。
   - **04 (API 契约)**：强制输出符合 OpenAPI 3.0 规范的 YAML，包含请求/响应的 `$ref` 定义。
3. **Skill 级约束融合**：全面引入 `jap-architect-guard` 的规则，在系统提示中硬性规定“跨文件命名一致性（如实体名在 02 和 04 中必须一致）”及“禁止废话解释”。

### 3.3 深度整合 MCP 与降级重试（确保 100% 成功率）

**方案**：在纯文本生成模式下，保留并强化 MCP 优先调用机制与智能重试，确保长文本/复杂文件绝对能出来。
**修改点**：
1. **MCP 优先策略**：
   - 01/02/03 若系统连接了图表 MCP（如 `mcp-diagram-server`），优先用其处理 Mermaid。
   - 02/04 若系统连接了数据库/API MCP，优先用其处理 DDL/OpenAPI。
2. **多策略降级重试 (Error Recovery)**：
   - **Attempt 1**：全量上下文（10000字）+ 完整 CoT 模板 + MCP 尝试。
   - **Attempt 2**：若超时或超 Token，自动切换为“精简上下文（6000字）+ 纯原生 LLM 生成”。
   - **Attempt 3**：若依然失败，触发“极简骨架生成”（只输出核心列表），确保流程绝不断裂。

### 3.4 传统文件名镜像导出（解决 P2 体验）

**方案**：满足用户“每次生成能看到最终成品文件”的心理预期。
**修改点**：`taskRoutes.ts` 中处理 `generate-next` 成功的逻辑。
**做法**：
- 当 `files/01.md` 生成成功落盘后，立刻拷贝/写入一份到 run 目录的根级，命名为 `01_产品功能脑图与用例.md`。
- 后续文件同理，确保本地文件树清晰可见。
