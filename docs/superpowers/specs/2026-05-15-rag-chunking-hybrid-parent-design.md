# RAG Chunking Hybrid Parent Design

## 背景

当前分块实现把父块按完整标题路径精确分组，导致深层结构化文档的 `parentContext` 过碎。

典型问题：

- `### 不写代码，只写 Spec` 的父上下文只包含标题下导语
- 后续 `步骤 1 / 要做什么 / 产出物` 被切到更深路径后，不再共享同一个父块
- 前端显示没有截断错误，问题来自后端父块聚合边界过细

目标是保留细粒度子块检索，同时把父上下文改成“更粗一级、可解释、够完整”的章节级上下文；对缺少标题结构的文章则退化为连续块簇。

## 设计目标

- 子块继续保持细粒度检索能力
- 结构化文档的父上下文按章节子树聚合，而不是按完整路径精确分组
- 非结构化文档退化为邻近块簇，不出现所有块共享一个空父块
- 保持现有检索、重排、注入主流程兼容

## 策略

### 1. 结构化文档

当文档存在足够多标题时，父块不再直接等于 `block.path`。

为每个块计算一个“父锚点路径”：

- 深层块优先挂到最近的较粗一级章节
- `h4/h5/h6` 优先挂到最近的 `h3`
- `h3` 优先挂到最近的 `h2`
- `h2` 优先挂到最近的 `h1`
- 如果没有可用上级，则回退到当前路径

结果：

- `步骤 1 -> 要做什么 / 产出物 / 示例` 共享同一个父块
- 子块本身仍保留原始 `path`
- `parentPath` 改为父锚点路径

### 2. 弱结构 / 无结构文档

当标题数量过少，或多数块没有稳定上级时，走簇模式：

- 按原始块顺序累积为父簇
- 单个父簇控制在较大的字符窗口内
- 新块加入后若超过阈值，则开启新父簇
- 子块继续从父簇内部做小块合并和大块切分

建议父簇窗口：

- `clusterTargetChars = 900`
- `clusterMaxChars = 1400`

这样无标题长文会形成多个连续父簇，而不是一个巨大空父块。

### 3. 子块策略

保留现有逻辑：

- `mergeSmallBlocks()`
- `splitOversizeBlocks()`

变化只发生在父块构建层：

- `path` 保留原始精细路径
- `parentPath` / `parentContext` 使用新的父块聚合结果
- `childIndexInParent` 仍按父块内部顺序计算

## 数据结构

在现有元数据上新增：

```ts
parentKind?: "section" | "cluster";
parentTitle?: string;
```

含义：

- `section`：按章节子树聚合出来的父块
- `cluster`：无结构文档退化出来的连续块簇
- `parentTitle`：父块展示标题；章节模式下取父章节名，簇模式下可取 `未命名片段 1` 之类的稳定标题

## 验收标准

- `不写代码，只写 Spec` 这类标题命中后，父上下文包含其下多个子标题内容，而不是只剩导语
- 结构化文档里同一大节下的细粒度块共享同一个更粗父块
- 无标题长文形成多个父簇，而不是全部块共享空 `parentPath`
- 现有 chunking 行为仍保留：标题不直接作为叶子检索块、跨章节不错误合并、小块合并和大块切分继续生效
- 检索、注入、前端展示不因字段扩展而出错

## 非目标

- 本轮不改 retrieval 打分逻辑
- 本轮不改 query rewrite / rerank / injection 主流程
- 本轮不引入语义分句模型或 LLM 辅助切块
