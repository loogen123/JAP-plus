# RAG 分块与检索改造设计

> 日期：2026-05-15
> 状态：待评审

## 1. 目标

- 解决当前分块缺少结构边界、切分随意的问题
- 解决当前检索对单文件设置硬上限，导致高相关结果被压制的问题

## 2. 范围

本次只调整两个模块：

- `src/rag/chunking/index.ts`
- `src/rag/retrieval/index.ts`

不改动：

- embedding 生成方式
- BM25 与向量召回的基本框架
- RRF 与 MMR 的整体流程
- 注入 prompt 的格式

## 3. 现状问题

### 3.1 分块问题

当前分块流程是：

1. 按 Markdown 标题切
2. 按段落切
3. 超长文本再按固定窗口切

问题：

- 结构粒度过粗，只识别标题和段落
- 列表、代码块、表格、引用等结构没有被显式建模
- 小块合并只看长度，不看结构归属
- 超长内容兜底时缺少明确的父结构信息

结果是分块边界不稳定，块内语义可能混杂，块间关系也不清晰。

### 3.2 检索问题

当前检索在融合与重排后，仍然执行“每个文档最多返回 2 个分块”的限制。

问题：

- 会压掉同一核心文档中的高置信度结果
- 排序目标从“全局最相关”变成“先按文档配额过滤”
- 当真正答案集中在一个文档中时，结果质量明显下降

## 4. 设计决策

### 4.1 结构化分块

分块改成三段式：

1. `parseBlocks`
2. `mergeSmallBlocks`
3. `splitOversizeBlocks`

核心原则：

- 先识别结构，再做长度控制
- 小块只在同一结构父节点内合并
- 超长块只作为兜底处理，不作为主切分方式

### 4.2 全局置信度排序

检索结果保留现有召回和融合流程，但移除“每文件最多 2 块”的硬限制。

最终结果规则：

- 先语义召回
- 再关键词召回
- 用 RRF 融合
- 用 MMR 做多样性重排
- 最终直接按全局结果截断到 `topK`

即：

- 不再按文件配额过滤
- 不再人为限制同一文档的命中数
- 以融合后的全局相关度为唯一主排序依据

## 5. 分块详细方案

### 5.1 结构识别

新增中间结构块概念，至少支持以下类型：

- `heading`
- `paragraph`
- `list`
- `table`
- `code`
- `quote`

每个结构块都带上基础元数据：

- `blockType`
- `sectionTitle`
- `path`
- `tokenCount`
- `startOffset`
- `endOffset`

其中：

- `sectionTitle` 表示最近的标题上下文
- `path` 表示结构路径，例如 `["系统设计", "检索流程"]`

### 5.2 parseBlocks

职责：

- 先按行扫描文本
- 识别标题、列表、代码块、表格、引用、普通段落
- 产出原始结构块序列

要求：

- 标题块只用于更新上下文，不单独作为最终检索块返回
- 相邻同类型且连续的行可以归并为同一个原始块
- 代码块必须整体保留，不能把代码和正文混切
- 表格按整体保留

### 5.3 mergeSmallBlocks

职责：

- 合并过短的结构块，减少碎片

规则：

- 只允许在同一 `path` 内合并
- 优先合并同类型块
- 不跨代码块与普通段落合并
- 不跨标题层级合并
- 如果前后都可合并，优先并入语义更接近的邻块；首版可直接优先并前块

### 5.4 splitOversizeBlocks

职责：

- 处理超过 `chunkSize` 的单个结构块

规则：

- 仅对超长 `paragraph` 或超长 `code` 做滑窗切分
- 滑窗切分后继承原始 `sectionTitle`、`path`、`blockType`
- 子块增加 `chunkIndex`，并保留原块范围信息

### 5.5 最终输出

`chunkText()` 仍返回 `ChunkDraft[]`，但元数据补充为：

- `docFileName`
- `chunkIndex`
- `tokenCount`
- `sectionTitle`
- `blockType`
- `path`
- `startOffset`
- `endOffset`

如果现有类型不完整，则扩展 `src/rag/types.ts` 中的 `ChunkMeta`。

## 6. 检索详细方案

### 6.1 保留现有主链路

继续保留：

- 向量检索或本地语义降级检索
- BM25 关键词检索
- `rrfFuse`
- `mmrRerank`

### 6.2 删除文档配额限制

删除：

- `MAX_RESULTS_PER_DOC`
- 结果组装阶段的 `perDocCount` 过滤逻辑

替换为：

- `mmrRerank` 结果直接按顺序取前 `topK`

### 6.3 排序原则

排序原则调整为：

- 是否返回某块，只由其全局相关度和重排结果决定
- 不再把“文档均匀分布”当作强约束

说明：

- 如果单一文档中连续多个块都高相关，就允许它们同时进入结果
- 如果担心重复，由 `MMR` 负责控制冗余，而不是用文档配额硬裁剪

## 7. 兼容性

- 对外接口 `chunkText()` 和 `retrieve()` 名称不变
- 现有知识库索引文件允许继续读取
- 新增元数据字段应设计为可选，避免旧数据立刻失效

如果旧索引缺少新元数据：

- 检索流程仍可工作
- UI 展示时按已有字段降级显示

## 8. 验收标准

### 8.1 分块

- Markdown 文档中的标题、列表、代码块、表格、引用可被稳定识别
- 小块不会跨章节乱合并
- 超长代码块和超长段落会被单独兜底切分

### 8.2 检索

- 检索结果不再受“每文件最多两块”限制
- 同一文档中多个高分块可同时进入结果
- 最终结果顺序与融合后的全局相关度一致

## 9. 实施顺序

1. 扩展 `ChunkMeta` 结构
2. 重写 `src/rag/chunking/index.ts` 的分块流程
3. 删除 `src/rag/retrieval/index.ts` 的文档配额逻辑
4. 补齐对应单测

## 10. 风险

- 结构化分块规则写得过细，会提高维护成本
- 偏弱格式文本可能无法完全识别结构，只能降级为段落块
- 删除文档配额后，结果更依赖 `MMR` 质量

对应策略：

- 首版只支持最核心结构类型
- 无法识别时统一回退为 `paragraph`
- 保留 `MMR`，不额外叠加新的硬规则
