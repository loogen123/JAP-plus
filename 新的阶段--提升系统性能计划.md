# 新的阶段--提升系统性能计划

## 0. 文档定位

- 阶段目标：在**不降低大模型输出质量**的前提下，提升 JAP-plus 的端到端流畅度（用户体感 + 程序运行时间）。
- 适用范围：`elicitation` 问卷/定稿链路 + `filewise` 01~08 生成链路 + 前端工作台交互链路。
- 本阶段约束：仅规划，不改业务功能定义，不牺牲 SDD/Gate 质量标准。

## 1. 依据与现状说明

- 文档依据：`项目模块说明.md`、`docs/sdd-plan.md`、`docs/sdd-vibe-coding-plan.md`。
- 代码依据：`src/services/taskService.ts`、`src/services/elicitationService.ts`、`src/tools/mcpClient.ts`、`src/controllers/taskController.ts`、`public/js/main.js`。
- Wiki 说明：仓库 `/wiki` 当前未提供独立内容（访问会回落到仓库主页），本计划以仓库内文档和代码为准。

## 2. 性能目标（不降质前提）

### 2.1 体验目标（用户可感知）

- 工作区状态刷新延迟（点击后看到状态变化）P95 < 800ms。
- 长任务期间界面无明显“卡死感”，日志区和文件预览可持续响应。
- 历史任务列表/SDD 源任务列表打开时间 P95 < 1.5s（100 个任务量级）。

### 2.2 运行目标（系统可度量）

- `GET /api/v1/tasks/filewise/:runId/events` 平均响应时延降低 60%+。
- SDD 阶段（08）总耗时降低 25%~40%（同等输入规模）。
- 全流程（01~08）总耗时降低 20%+。
- 文件系统 I/O 次数（每次生成）降低 30%+。

### 2.3 质量红线

- 不降模型规格，不降 prompt 约束，不减少 SDD 章节，不放宽 Gate 校验规则。
- 优化仅通过：并行化、缓存、增量更新、I/O 减载、前后端协同刷新策略实现。

## 3. 瓶颈地图（当前）

1. 事件读取是“全量读文件再截尾”  
   `readRunEventsTail()` 每次读取完整 `events.log` 再切片，日志增长后会显著变慢。

2. 前端刷新链路存在重复请求  
   `refreshFilewiseRun()` 每次都追加 `pullRecentEvents()`，且多个 WS 事件会频繁触发刷新。

3. 每次刷新都回传完整 `currentFileContent`  
   大文件（06 HTML、08 SDD）会被反复传输与重渲染。

4. SDD（08）生成中存在重复文件读取  
   `loadSddEvidence()`、`runLocalSddPrecheck()`、`validateSddGate()`多次重复读取 02/03/04/01~07。

5. 已审批摘要反复重算  
   `loadApprovedArtifactSummary()` 每次尝试都顺序读取已审批文件全文。

6. 历史列表与 SDD 源列表为重 I/O 扫描  
   `listHistoryRecords()`、`listSddSourceRuns()`在目录多时会明显拖慢。

7. MCP 工具发现重复调用  
   `callToolByCandidateNames()` 每次都 `listTools()`，缺少稳定缓存策略。

8. 前端日志无限追加  
   `addLog()` 不设上限，长会话会带来 DOM 增长和渲染压力。

9. 背景定稿频繁触发  
   问卷作答过程中的 `runBackgroundFinalize()` 仍会触发较重定稿调用，存在重复计算。

10. 运行态并发保护弱  
    同一 run 在高频操作下可能出现竞争（状态被交错覆盖）。

## 4. 分阶段执行计划

## 阶段 A：可观测性与基线建立（先量化，再优化）

### 目标

- 建立“每条链路耗时画像”，避免盲目优化。

### 计划

- 为核心接口打点：入参规模、I/O 时长、LLM 时长、MCP 时长、总时长。
- 建立 run 级耗时账本（01~08 各文件：准备/调用/写入/校验）。
- 新增前端性能日志：刷新频率、渲染耗时、事件堆积情况。

### 验收

- 可以给出：慢请求 TopN、慢阶段 TopN、慢文件 TopN。

## 阶段 B：I/O 与数据通路减载（优先收益最大）

### 目标

- 降低磁盘读写放大和网络重复传输。

### 计划

- 事件流改为增量读取（cursor/offset），避免每次全量读 `events.log`。
- `meta.json` 读写引入短生命周期内存缓存 + 写合并（保持落盘一致性）。
- `GET /filewise/:runId` 改为可选字段返回：默认不带全文，按需拉取文件内容。
- `readPreview()` 改为“仅读取前 N 字节”，避免整文件读。
- 历史列表与 SDD 源列表改为“两段式加载”：先元数据，后按需校验正文完整性。

### 验收

- 大日志场景下事件接口耗时显著下降，历史列表加载稳定在目标范围内。

## 阶段 C：08 SDD 生成链路专项提速（不降质）

### 目标

- 在保持约束严格度和输出质量的前提下缩短 08 阶段耗时。

### 计划

- 统一构建 `08` 的“输入快照对象”（evidence/openapi/domain/state/qa/skill），单次生成复用，避免重复读取。
- 预检与文本准备并行执行（文件读取并发化，限制并发度防止抖动）。
- `loadApprovedArtifactSummary()` 增量缓存：仅在文件状态变更时重算。
- Gate 维持严格，但减少重复 prompt 序列化和重复字符串截断计算。

### 验收

- 08 阶段总耗时下降 25%+，冲突检出能力不下降。

## 阶段 D：前端流畅度专项优化

### 目标

- 降低“明明在跑但界面卡”的感知问题。

### 计划

- 刷新去抖：WS 高频事件合并（如 300~800ms 批处理）。
- `refreshFilewiseRun()` 与 `pullRecentEvents()` 解耦，避免每次都双请求。
- 日志列表上限（如 500~1000 条）+ 自动裁剪 + “查看完整日志”入口。
- 用户编辑 `filePreview` 时启用“脏标记”，避免后台刷新覆盖输入。
- 仅在必要时弹窗（避免状态抖动频繁开关审阅窗口）。

### 验收

- 长任务（>10 分钟）期间页面交互仍顺畅，无明显输入卡顿。

## 阶段 E：并发安全与稳定性

### 目标

- 防止 run 级并发请求导致状态竞争和无效重试。

### 计划

- 引入 run 级互斥锁（generate/approve/reject/save-edit 串行化）。
- 为幂等操作增加 requestId（防重放/防重复提交）。
- 对背景 finalize 引入可取消机制（新请求取消旧请求，结果只认最新版本）。

### 验收

- 并发点击与弱网重试场景下，状态机仍保持一致，不出现“回跳/错位”。

## 5. 优先级与排期建议

1. 第 1 周：阶段 A + 阶段 B（事件/历史/响应载荷优化）。  
2. 第 2 周：阶段 C（08 链路专项，重点降时）。  
3. 第 3 周：阶段 D + 阶段 E（前端体感和并发稳定性收口）。  

> 说明：如资源紧张，优先顺序固定为 **B > C > D > E**，因为 B/C 对总耗时收益最大。

## 6. 验收指标（阶段完成判定）

- 指标 1：全流程（01~08）平均耗时下降 >= 20%。
- 指标 2：08 阶段平均耗时下降 >= 25%。
- 指标 3：事件接口在大日志场景的 P95 降低 >= 60%。
- 指标 4：前端长会话（30 分钟）无明显卡顿，日志渲染稳定。
- 指标 5：输出质量抽检通过率不低于当前基线（章节完整性、命名一致性、Gate 冲突识别）。

## 7. 风险与应对

- 风险：缓存导致旧数据被误用。  
  应对：基于文件 `mtime` + run 版本号失效；关键路径保留强制刷新开关。

- 风险：并行化后引入竞态。  
  应对：run 级锁 + 幂等令牌 + 状态转移前后断言。

- 风险：前端去抖导致“看起来不实时”。  
  应对：关键事件（失败/完成）仍即时渲染，普通日志批量渲染。

---

## 8. 结论

本阶段不改业务目标、不降模型质量，专注通过工程手段提升性能。  
执行完成后，系统应从“可用”升级到“高频可用、长会话稳定、生成链路更快”的工程化状态。
